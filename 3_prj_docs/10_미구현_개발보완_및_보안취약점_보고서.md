# 미구현 사항, 개발 보완 및 보안 취약점 보고서

본 문서는 현재 시스템의 한계점과 향후 기술적으로 보완이 필요한 사항, 그리고 잠재적인 보안 취약점을 체계적으로 정리하여 향후 유지보수 및 고도화의 지침으로 활용합니다.

## 1. 미구현 사항 (Unimplemented Features)

현재 요구사항 범위(Scope) 밖이거나 시간적 제약으로 구현되지 않은 기능입니다.

### 1.1 자동 채점 및 결과 피드백 시스템
- **현황**: 응시자가 구글 폼에 답안을 입력하면 수동으로 확인해야 함. (요구사항 고도화 논의 대상)
- **필요성**: 문제 풀이 즉시 정답 여부를 확인하고 해설로 연결되는 자동화 루프 필요.

### 1.2 사용자별 학습 진도 및 성적 관리
- **현황**: 현재는 '인증 코드' 기반의 단순 입장 방식으로, 과거 학습 이력이나 오답 노트를 관리할 수 없음. (요구사항 제외 사항이나 향후 확장 고려 대상)
- **필요성**: 재접속 시 이전에 풀었던 문항 표시 및 성취도 그래프 제공.

### 1.3 순위표(Leaderboard) 및 전체 통계 리포트
- **현황**: 전체 응시자 간의 비교 데이터나 평균 통계 기능 부재.
- **필요성**: 수험생들에게 본인의 위치를 알리고 학습 동기를 부여하기 위한 통계 데이터 시각화.

### 1.4 계정 기반 인증 시스템으로의 전환
- **현황**: DB 없는 단순 난수 코드 비교 방식 (저비용 운영 집중).
- **필요성**: 독자 관리 및 마케팅 데이터 확보를 위해 향후 가입 기반 시스템으로의 확장 유연성 확보 필요.

### 1.5 전용 모바일 앱 또는 프로그레시브 웹 앱(PWA)
- **현황**: 모바일 접속 시 PC 안내 페이지만 제공.
- **필요성**: 문제 읽기나 가벼운 이론 학습은 모바일에서도 최적화된 경험 제공 필요.

## 2. 기술적 보완 필요 사항 (Technical Improvements)

개발 구조적 안정성과 유지보수성 향상을 위한 보완점입니다.

### 2.1 에러 로깅 및 모니터링 시스템 부재
- **현황**: API 호출 실패나 인증 오류 발생 시 서버 콘솔 로그에만 기록됨.
- **보완**: Sentry나 LogRocket 등을 도입하여 실사용자 환경에서 발생하는 런타임 에러를 실시간 감지 필요.

### 2.2 클라이언트 상태 관리 최적화
- **현황**: 대시보드 등에서 `useState`와 `useEffect`를 사용한 로컬 캐싱에 의존.
- **보완**: TanStack Query(React Query) 등을 도입하여 서버 상태 동기화 및 데이터 무결성 강화.

### 2.3 통합 테스트 코드 부재
- **현황**: 모든 검증을 수동 테스트에 의존하고 있음.
- **보완**: Jest(Unit Test) 및 Playwright(E2E Test)를 도입하여 코드 수정 시 기존 기능(인증, 캐시 갱신 등)의 영향도를 자동 체크해야 함.

## 3. 잠재적 보안 취약점 (Security Vulnerabilities)

현재의 간소화된 아키텍처에서 발생할 수 있는 보안적 위험 요소입니다.

### 3.1 클라이언트 측 인증 우회 가능성
- **위험**: `auth_token` 쿠키를 브라우저 개발자 도구에서 수동으로 생성하면 대시보드 진입이 가능할 수 있음.
- **보완**: 서버 사이드(Middleware)에서 실제 인증 코드가 유효한지 매번 검증하거나, JWT(JSON Web Token)와 같은 서명된 토큰 방식을 도입해야 함.

### 3.2 무차별 대입 공격(Brute-force) 취약성
- **위험**: 16자리 인증 코드를 무작위로 입력하여 통과될 확률이 존재(비록 확률은 낮으나).
- **보완**: 일정 횟수 이상 인증 실패 시 해당 IP를 한시적으로 차단하거나 캡차(reCAPTCHA)를 도입하여 자동화된 공격을 방어해야 함.

### 3.3 GAS API URL 노출 (조치 완료 ✅)
- **위험**: `NEXT_PUBLIC_GAS_API_URL`은 브라우저 소스 코드에 노출될 위험이 있었음.
- **조치**: 
    1. `/api/exams` 프록시 API Route를 생성하여 클라이언트에서 GAS 직접 호출을 차단함.
    2. 서버 환경 변수(`GAS_API_URL`)를 사용하여 실제 GAS 주소를 서버 사이드에서만 관리하도록 강화함.
    3. `getExams` 로직을 수정하여 클라이언트에서는 프록시를 통해서만 데이터를 가져오도록 변경함.

### 3.4 개방형 리디렉션(Open Redirect) 위험 (조치 완료 ✅)
- **위험**: `/auth?redirect=...` 파라미터를 통해 외부 악성 사이트로의 리디렉션 유도 가능성.
- **조치**: 
    1. `app/auth/page.tsx`에서 리디렉션 경로 검증 로직 추가.
    2. `/`로 시작하는 내부 상대 경로만 허용하고, 외부 URL(http/https)은 모두 `/dashboard`로 강제 고정함.
